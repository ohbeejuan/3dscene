# CS 330 - Computational Graphics and Visualization Portfolio

## How do I approach designing software?

Working on this project helped me develop skills in planning visual layouts and interactive systems before writing code. I learned to think about how individual components — bricks, circles, paddles, and power-ups — would interact as a whole system, and to design each piece with clear responsibilities. My design process started with understanding the starter code's structure, then identifying what each rubric criterion required, and finally planning enhancements that would build on each other logically. For example, I designed the brick layout with graduated difficulty in mind from the start, which meant choosing colors, sizes, and hit points that would create a cohesive visual hierarchy. I also learned to anticipate how new features like multi-ball power-ups could conflict with existing systems like circle-to-circle collision, which taught me to design with edge cases in mind. These tactics — modular planning, thinking about system interactions, and designing for extensibility — are directly applicable to any future software project, whether it involves graphics or not.

## How do I approach developing programs?

My primary development strategy was iterative building and testing. Rather than writing all features at once, I started with the base animation, then layered on enhancements one at a time: first the brick layout and paddle, then physics-based movement, then collision behaviors, then power-ups, and finally the title and game over screens. Iteration was essential throughout — for instance, my initial circle-to-circle collision caused multi-ball power-ups to self-destruct on spawn, so I iterated by adding a spawn invulnerability timer to fix the conflict. Similarly, the paddle collision initially used custom angle-adjustment logic that felt unnatural, so I iterated back to using the same AABB reflection physics as the bricks. My approach evolved from making broad structural changes early in development to making precise, targeted adjustments later. By the final milestones, I was comfortable reading through the full codebase, identifying exactly where a change needed to happen, and implementing it without disrupting existing functionality. This progression from large-scale to fine-grained development mirrors how real software projects mature.

## How can computer science help me in reaching my goals?

Computational graphics gave me hands-on experience with concepts that bridge multiple areas of computer science: real-time rendering loops, collision detection algorithms, state management, and object-oriented design. These skills translate directly into future coursework involving simulations, game development, data visualization, or any application with a visual interface. Professionally, understanding how graphics pipelines work, how to manage frame-by-frame updates, and how to structure interactive applications gives me a foundation for roles in software engineering, game development, UI/UX development, or any field where visual computing is relevant. Even beyond graphics-specific careers, the problem-solving patterns I practiced — decomposing complex interactions into manageable classes, debugging visual behavior by reasoning about coordinate systems and timing, and balancing performance with visual quality — are skills that strengthen my ability to tackle any complex programming challenge.
